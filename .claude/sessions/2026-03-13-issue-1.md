## Reference: svelte-table-views-sidebar (~/Desktop/svelte-table-views-sidebar)

### What it is
- Standalone npm package (`svelte-table-views-sidebar` v0.1.2)
- Single component: `ViewSidebar.svelte`
- Peer dep on `svelte-table-views-tanstack` (optional)
- Pure UI â€” no persistence, no store management

### Component API
- **Props**: `views: SidebarView[]`, `groups: ViewGroup[]`, `selectedViewId`, `storageKey`, `isDocked`, `width`, `showSearch`, `showPinned`, `searchPlaceholder`
- **Events**: `select`, `pin`, `groupToggle`
- **Features**: search/filter, pinned views section, collapsible groups, localStorage state persistence (collapsed/pinned/docked/width), keyboard nav, CSS custom properties theming
- **Types**: `SidebarView`, `ViewGroup`, `ViewSidebarProps`, `SidebarState`, event types

### Key difference from ViewSelector
- ViewSelector = dropdown (click to open, auto-closes)
- ViewSidebar = persistent always-visible panel (flex layout alongside content)
- Sidebar adds: groups, pinned views, dockable/width, richer UI
- Sidebar is purely presentational â€” consumer maps their views to `SidebarView[]`

### Integration pattern (in tanstack world)
```svelte
$: sidebarViews = $savedViews.map(v => ({
  id: v.id, name: v.name, description: v.description,
  groupId: v.groupId, isDefault: v.isDefault
}));
<ViewSidebar views={sidebarViews} {groups} on:select={handleSelect} />
```

## Decision: Integrate or keep separate?

### Option A: Integrate into svelte-gridlite-views
- **Pro**: Single package for consumers, shared ViewStoreBundle prop pattern, consistent styling
- **Pro**: Can directly accept `viewStore: ViewStoreBundle` prop (like ViewSelector) â€” no mapping boilerplate
- **Pro**: Can access store internals (activeViewId, activeViewModified, actions) directly
- **Con**: Increases package size (~5KB)

### Option B: Keep separate package (svelte-gridlite-sidebar or similar)
- **Pro**: Lighter core package, optional install
- **Con**: Extra dep for consumers, needs peer dep on svelte-gridlite-views
- **Con**: Consumer must manually map stores â†’ SidebarView[] (boilerplate)
- **Con**: Two packages to version/publish/maintain for a small library
